# Tesla Earnings RAG + Agent

A Python RAG (Retrieval-Augmented Generation) project with a LangGraph agent that answers
questions about a Tesla earnings call transcript. The app embeds a PDF into Chroma,
retrieves relevant chunks with Amazon Titan on AWS Bedrock, generates an answer with
Claude, and falls back to Tavily web search when the transcript alone cannot answer
the question.

## Features

- **Document RAG** — PDF ingest, chunking, Titan embeddings, Chroma vector search
- **Grounded answers** — Claude on Bedrock answers from retrieved context only
- **Hybrid agent** — LangGraph routes to web search when RAG does not answer the question
- **Persistent index** — Chroma reuses embeddings on later runs

## What It Uses

- Python
- [LangChain](https://python.langchain.com/) — document loading, splitting, embeddings, chat
- [LangGraph](https://langchain-ai.github.io/langgraph/) — agent orchestration and conditional routing
- [Tavily](https://tavily.com/) — web search fallback
- `langchain-community` — `PyPDFLoader` for PDF ingestion
- `langchain-text-splitters` — `RecursiveCharacterTextSplitter` for chunking
- `langchain-aws` — `BedrockEmbeddings` and `ChatBedrockConverse`
- ChromaDB — local vector storage
- Amazon Titan Text Embeddings V2 — embeddings
- Anthropic Claude on AWS Bedrock — generation and routing decisions
- Cosine distance for vector search

## Architecture

```text
User question
     │
     ▼
┌─────────────────────────────────────┐
│  RAG pipeline                       │
│  embed query → Chroma top-k →       │
│  Claude answer from context         │
└─────────────────────────────────────┘
     │
     ▼
┌─────────────────────────────────────┐
│  LangGraph agent                    │
│                                     │
│  in_transcript (judge RAG answer)   │
│           │                         │
│     ┌─────┴─────┐                   │
│     ▼           ▼                   │
│  answered   needs web search        │
│     │           │                   │
│     ▼           ▼                   │
│ rag_answered  search_web (Tavily)   │
│     │           │                   │
│     │           ▼                   │
│     │      final_answer (Claude)    │
│     │           │                   │
│     └─────┬─────┘                   │
│           ▼                         │
│        final reply                  │
└─────────────────────────────────────┘
```

This pattern works well for questions the transcript can answer directly (e.g. earnings
topics discussed on the call) and questions that need information after the report
(e.g. Robotaxi progress since the call).

## Setup

Create and activate a virtual environment:

```bash
python -m venv .venv
source .venv/bin/activate
```

Install dependencies:

```bash
pip install -r requirements.txt
```

If you set up the project before LangGraph and Tavily were added, install the extra packages:

```bash
pip install langgraph langchain-aws langchain-community langchain-text-splitters tavily-python python-dotenv
```

## Environment Variables

### AWS Bedrock

Configure AWS credentials:

```bash
aws configure
```

The app uses these Bedrock models (in `src/rag_app/config.py`):

```python
MODEL_ID = "us.anthropic.claude-sonnet-4-5-20250929-v1:0"
EMBEDDING_MODEL_ID = "amazon.titan-embed-text-v2:0"
EMBEDDING_DIMENSIONS = 1024
```

Enable both models in the AWS Bedrock console. To check available inference profiles:

```bash
aws bedrock list-inference-profiles --region us-east-1
```

### Tavily (web search fallback)

Sign up at [tavily.com](https://tavily.com) and set your API key:

```bash
export TAVILY_API_KEY="tvly-..."
```

Or create a `.env` file in the project root:

```text
TAVILY_API_KEY=tvly-...
```

Web search is only used when the agent decides the RAG answer did not sufficiently
answer the question.

## How It Works

### Indexing (first run)

1. Load `./doc/Tesla_earnings.pdf` with LangChain `PyPDFLoader`
2. Split text into chunks (`chunk_size=1000`, `chunk_overlap=120`)
3. Embed each chunk with `BedrockEmbeddings` (Titan Text Embeddings V2)
4. Store chunks and embeddings in `./db/Tesla_earnings_titan_db`

Later runs reuse the existing Chroma collection.

### Query + agent (every run)

1. Embed the question and retrieve the top matching chunks from Chroma
2. Build a prompt with retrieved context and ask Claude for a RAG answer
3. Run the LangGraph agent:
   - **Judge** — Claude decides if the RAG answer actually answered the question
   - **RAG path** — return the transcript-grounded answer
   - **Web path** — Tavily search → Claude synthesizes an answer from web results

The default question (in `config.py`) is:

```text
Has the robotaxi made in progress since thie report came out?
```

Change `DEFAULT_QUERY` in `src/rag_app/config.py` to try other questions.

### Module overview

| Module | Role |
|--------|------|
| `config.py` | Paths, model IDs, default query |
| `vector_store.py` | Chroma collection setup, ingestion, retrieval |
| `pipeline.py` | PDF indexing, RAG retrieval, LangGraph agent graph |
| `app.py` | Entry point |

### Agent state

The LangGraph agent passes a shared state object between nodes:

| Field | Purpose |
|-------|---------|
| `question` | User question |
| `rag_answer` | Answer from transcript RAG |
| `use_web_search` | Whether to fall back to web search |
| `web_results` | Snippets from Tavily |
| `answer` | Final response |

## Reset the Vector Database

To re-chunk and re-embed the PDF (e.g. after changing chunk settings):

```bash
rm -rf ./db/Tesla_earnings_titan_db
python app.py
```

## Tuning Notes

- **Chunk size** — Larger chunks (e.g. 1000 chars) carry more context than very small
  chunks (e.g. 100 chars), which tend to produce sentence fragments and weaker retrieval.
- **Top-k** — `n_results=3` in `pipeline.py` controls how many chunks are sent to the LLM.
  Increase k if answers miss relevant passages split across chunks.
- **Cosine search** — Chroma uses `hnsw:space: cosine`. Titan embeddings use
  `normalize: True`. Similarity can be approximated as `1 - distance`.
- **Re-index after chunk changes** — Updating `chunk_size` or `chunk_overlap` does not
  change an existing collection; delete `db/` or use a new collection name to rebuild.
